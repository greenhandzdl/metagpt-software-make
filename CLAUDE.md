## 八大智能体角色

所有角色实现位于 `meta/roles/`，每个角色遵循单一职责原则：

### 开发实现类
- **前端开发** — 实现界面与交互逻辑
- **后端开发** — 实现业务逻辑与 API
- **数据库工程师** — 设计表结构、索引、查询优化、数据迁移
- **容器编排/平台工程师** — 编写 Dockerfile、K8s 配置、CI/CD 流水线

### 质量与安全类
- **测试工程师** — 编写单元测试、集成测试、端到端测试，执行自动化回归
- **代码审查者** — 检查代码规范、逻辑缺陷、设计合理性（可由资深开发兼任）
- **安全审计员** — 静态分析、依赖扫描、秘密检测、权限模型验证

### 协调与指导类
- **架构师/技术负责人** — 制定编码规范、安全策略、质量门禁，裁决技术债务与安全修复优先级

## AGENTS 工作流

1. 开始任务前，读取 `AGENTS/plan.md` 了解当前进度
2. 遇到多角色冲突时，查阅 `AGENTS/conflict.md` 按优先级裁决
3. 完成复杂问题后，将成功经验记录到 `AGENTS/experience.md`
4. 每次完成任务后更新 `AGENTS/plan.md` 标记进度

## 冲突处理优先级（conflict.md 核心规则）

1. **安全审计员** — 安全修复最高优先级，不可被其他需求阻塞
2. **架构师/技术负责人** — 架构决策和技术规范可否决实现方案
3. **代码审查者** — 代码质量门禁，不达标不可合并
4. **测试工程师** — 测试覆盖率和通过率是合并前置条件
5. 其余角色按任务依赖链自然排序

## 开发流程

1. 架构师输出技术方案和编码规范
2. 数据库工程师设计数据模型和迁移脚本
3. 前后端开发并行实现功能
4. 容器工程师编写部署配置
5. 测试工程师编写并执行测试
6. 代码审查者和安全审计员进行审查
7. 架构师做最终裁决后合并
